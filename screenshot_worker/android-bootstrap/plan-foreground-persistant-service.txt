SECTION 2: FIX SERVICE START BEHAVIOR (ScreenshotService.kt)
================================================================================

CURRENT BUG:
    Line 97: flags = START_STICKY
    Line 99: return super.onStartCommand(intent, flags, startId)

    This does NOT return START_STICKY. super.onStartCommand() in Service class
    returns START_STICKY_COMPATIBILITY by default. The assignment to `flags` is
    a no-op because `flags` is a method parameter (local var) and does not
    affect the return value.

REQUIRED CHANGE:
    Replace the return statement with an explicit return value:

    return START_REDELIVER_INTENT

    Why START_REDELIVER_INTENT instead of START_STICKY?
    - MediaProjection requires the original Intent with resultCode and data
    - START_STICKY would restart with a null intent, which cannot recreate
      the MediaProjection (the permission token dies with the process)
    - START_REDELIVER_INTENT redelivers the last intent, giving us a chance
      to recover if the OS killed us but the token is still valid

    However, MediaProjection tokens DO NOT survive process death. If the app
    is fully killed by the OS, neither START_STICKY nor START_REDELIVER_INTENT
    can resurrect the capture. The user MUST re-open the app and re-grant
    permission. This is an Android security limitation we cannot bypass.

REFERENCE PATTERN (from MyService2.java, ScreenshotService.java, RecPlayService.java):

3b. In ScreenshotService.onDestroy(), before super.onDestroy(), add:

    override fun onDestroy() {
        logger.i(TAG, "Service destroying — scheduling restart")

        // Notify that we need restart
        val restartIntent = Intent(ACTION_RESTART_SERVICE).apply {
            setPackage(packageName)
        }
        sendBroadcast(restartIntent)

        // ... existing cleanup code ...
        captureJob?.cancel()
        serviceScope.cancel()
        transport?.stop()
        virtualDisplay?.release()
        imageReader?.close()
        mediaProjection?.stop()
        releaseWakeLock()

        super.onDestroy()
    }

3c. Update BootReceiver to handle ACTION_RESTART_SERVICE:

    class BootReceiver : BroadcastReceiver() {
        override fun onReceive(context: Context, intent: Intent) {
            when (intent.action) {
                Intent.ACTION_BOOT_COMPLETED -> {
                    // Cannot auto-start MediaProjection service without user interaction.
                    // Show a notification prompting user to open app.
                    showBootNotification(context)
                }
                ScreenshotService.ACTION_RESTART_SERVICE -> {
                    // Service was killed by OS. We cannot restart capture automatically
                    // because MediaProjection token is dead. Show recovery notification.
                    showRecoveryNotification(context)
                }
            }
        }
    }

    NOTE: The BootReceiver must be registered in AndroidManifest with BOTH
    intent filters: BOOT_COMPLETED and the custom ACTION_RESTART_SERVICE.

3d. Update AndroidManifest.xml BootReceiver to remove the restrictive
    android:permission attribute that blocks custom broadcasts:

    CURRENT (broken for custom broadcasts):
        android:permission="android.permission.RECEIVE_BOOT_COMPLETED"

    FIX: Remove android:permission from the receiver tag entirely, or use
    two separate <receiver> declarations. The permission attribute prevents
    ANY sender without that permission from triggering the receiver — including
    your own app sending custom restart broadcasts.

    CORRECT:
        <receiver
            android:name=".service.BootReceiver"
            android:enabled="true"
            android:exported="true">
            <intent-filter>
                <action android:name="android.intent.action.BOOT_COMPLETED" />
            </intent-filter>
            <intent-filter>
                <action android:name="com.openmule.screenshotworker.RESTART_SERVICE" />
            </intent-filter>
        </receiver>

SECTION 4: ALARM MANAGER PERIODIC HEALTH CHECK
================================================================================

CURRENT STATE:
    No mechanism exists to periodically verify the service is running and
    restart it if needed.

REFERENCE PATTERN (from MainActivity.java revive()):
    Uses AlarmManager.setExactAndAllowWhileIdle() with a PendingIntent that
    starts a foreground service, even in Doze mode.

REQUIRED CHANGES:

4a. Create a new file: ServiceKeepAliveHelper.kt

    object ServiceKeepAliveHelper {
        private const val ALARM_REQUEST_CODE = 9876
        private const val INTERVAL_MS = 60_000L  // 1 minute

        fun schedule(context: Context) {
            val alarmManager = context.getSystemService(Context.ALARM_SERVICE) as AlarmManager
            val intent = Intent(context, ScreenshotService::class.java)
            val pendingIntent = PendingIntent.getForegroundService(
                context,
                ALARM_REQUEST_CODE,
                intent,
                PendingIntent.FLAG_UPDATE_CURRENT or PendingIntent.FLAG_IMMUTABLE
            )

            // For API 23+, use setExactAndAllowWhileIdle to bypass Doze
            if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.M) {
                alarmManager.setExactAndAllowWhileIdle(
                    AlarmManager.ELAPSED_REALTIME_WAKEUP,
                    SystemClock.elapsedRealtime() + INTERVAL_MS,
                    pendingIntent
                )
            } else {
                alarmManager.setExact(
                    AlarmManager.ELAPSED_REALTIME_WAKEUP,
                    SystemClock.elapsedRealtime() + INTERVAL_MS,
                    pendingIntent
                )
            }
        }

        fun cancel(context: Context) {
            val alarmManager = context.getSystemService(Context.ALARM_SERVICE) as AlarmManager
            val intent = Intent(context, ScreenshotService::class.java)
            val pendingIntent = PendingIntent.getForegroundService(
                context,
                ALARM_REQUEST_CODE,
                intent,
                PendingIntent.FLAG_UPDATE_CURRENT or PendingIntent.FLAG_IMMUTABLE
            )
            alarmManager.cancel(pendingIntent)
        }
    }

4b. Call ServiceKeepAliveHelper.schedule() in ScreenshotService.onStartCommand()
    when successfully started, and cancel in onDestroy().

    IMPORTANT: The alarm fires even if the service is already running. In
    onStartCommand(), check if mediaProjection != null and if so, just
    call startForeground() again to satisfy the foreground requirement and
    return START_REDELIVER_INTENT. Do not try to reinitialize capture.

SECTION 11: KNOWN LIMITATIONS (cannot be worked around)
================================================================================

1. MediaProjection permission tokens are tied to the process lifetime.
   If the OS kills the app process, the token is invalidated. The user MUST
   reopen the app and tap Start to re-grant permission. There is no API
   to persist or recreate a MediaProjection without user interaction.

2. On Android 12+ (API 31+), foreground services started from the background
   (e.g., BootReceiver on BOOT_COMPLETED) are heavily restricted. The
   BootReceiver cannot directly start ScreenshotService. It can only show
   a notification asking the user to open the app.

3. Some OEMs (Xiaomi, Oppo, Vivo, Huawei) maintain their own kill lists
   independent of Android's standard behavior. Even with all protections,
   the app may be killed. The only reliable workaround is:
   - Disable battery optimization
   - Enable auto-start in OEM security app
   - Lock the app in recents
   - Add to "never sleep" / "protected apps" list
